Las capacidades de aislamiento y sandboxing de Piston ya se describieron en la Sección~\ref{SS:DI_TEC_MOTOR}. Este apartado se centra exclusivamente en cómo se integra ese motor en el flujo de la aplicación desde que el alumno pulsa el botón de ejecución hasta que el resultado aparece en la terminal.

\paragraph{Flujo de datos end-to-end.}

El alumno escribe código en el editor Monaco y pulsa el botón de ejecutar. Vue envía una petición POST autenticada al \textit{endpoint} \texttt{/compiler/execute/} de Django con el código fuente, el lenguaje y la versión del \textit{runtime}. La vista \texttt{execute\_code} de la aplicación \texttt{apps/compiler} construye el \textit{payload} conforme al contrato de la API REST interna de Piston y lo reenvía vía HTTP:

La URL del servicio se lee de la variable de entorno \texttt{PISTON\_URL}, lo que permite apuntar a distintas instancias sin modificar el código. Piston levanta el contenedor efímero, ejecuta el programa y devuelve un objeto JSON con los campos \texttt{compile} (errores de compilación para lenguajes como C o Java) y \texttt{run} (salida estándar y código de salida). Django extrae ambos campos y los normaliza:

El campo \texttt{status} de Piston codifica el motivo de finalización del proceso: \texttt{TO} indica \textit{timeout}, \texttt{CE} error de compilación, \texttt{SG} señal del sistema operativo (típicamente desbordamiento de memoria o fallo de segmentación), \texttt{OL} salida demasiado larga, entre otros. Django mapea cada código a un mensaje en español orientado al alumno que se concatena al \texttt{stderr} antes de enviarlo al \textit{frontend}, de modo que el alumno recibe una explicación comprensible en lugar de un código opaco.

\paragraph{Cierre del ciclo con el tutor.}

La respuesta normalizada \texttt{\{stdout, stderr, exit\_code, language\}} llega al \textit{frontend} y se muestra en el panel de salida integrado. Pinia almacena la última salida en el estado del editor; cuando el alumno envía el siguiente mensaje al tutor, ese valor se incluye automáticamente en el campo \texttt{last\_output} del \textit{payload} del chat. Django lo inyecta como segundo mensaje de sistema en el contexto del LLM (Sección~\ref{SS:DI_IMP_PROMPTS}), cerrando el ciclo en el que el tutor puede razonar sobre el comportamiento real del código del alumno sin que este tenga que copiar y pegar ninguna traza de error.
