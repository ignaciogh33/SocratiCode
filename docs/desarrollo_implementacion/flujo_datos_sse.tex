La naturaleza autorregresiva de los LLMs hace que una respuesta extensa pueda tardar varios segundos en generarse. Esperar a que Ollama termine por completo para entonces enviar el bloque de texto al cliente produciría una pantalla congelada durante ese tiempo, algo inaceptable en un entorno de tutoría interactiva. La solución es abrir una conexión de larga duración y enviar cada fragmento de texto (\textit{token}) en el instante exacto en que el modelo lo produce, de modo que el alumno ve la respuesta construirse letra a letra. Como se analizó en el estado del arte (Sección~\ref{SS:EA_ARQ_PROTOCOLOS}), esta comunicación puede implementarse con WebSockets o con SSE; a continuación se detalla la elección y su implementación concreta.

\paragraph{Por qué SSE y no WebSockets.}

El flujo de datos del tutor socrático es estrictamente unidireccional durante la generación de una respuesta: el servidor empuja \textit{tokens} al cliente, y el cliente no necesita enviar nada hasta que el stream ha terminado. Abrir un canal WebSocket bidireccional \textit{full-duplex} habría introducido una complejidad innecesaria: requiere un \textit{protocol upgrade} explícito (de HTTP a WS), implica mantener el estado de la conexión en el servidor y añade una capa de gestión de reconexión manual. Server-Sent Events, en cambio, utiliza una conexión HTTP estándar de larga duración con el tipo de contenido \texttt{text/event-stream}; el navegador la maneja de forma nativa y el servidor puede cerrarse cuando el stream finaliza sin necesidad de ningún protocolo adicional.

En el lado del cliente, la API estándar del navegador \texttt{EventSource} no es adecuada aquí porque solo admite peticiones GET, mientras que el envío del mensaje del alumno requiere un POST con cuerpo JSON. El \texttt{TextDecoder} convierte los bytes entrantes en texto UTF-8 con el flag \texttt{stream: true}, que le indica que los datos aún no están completos y que no debe descartar secuencias de bytes incompletas al final de cada chunk. El \textit{buffer} acumula el texto recibido y lo divide por líneas; la última línea (potencialmente incompleta si el chunk llegó partido por la red) se retiene en el buffer para el siguiente ciclo.
