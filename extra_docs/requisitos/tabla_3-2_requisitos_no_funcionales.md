# Tabla 3-2: Requisitos No Funcionales de SocratiCode

## Rendimiento

| ID     | Nombre                               | Descripción                                                                                                                                                                                                                                  | Métrica / Restricción                |
|--------|--------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|--------------------------------------|
| RNF-01 | Latencia de streaming                | La respuesta del tutor debe comenzar a llegar al navegador (primer token SSE) antes de que el usuario perciba una espera excesiva. El streaming token a token minimiza el *time-to-first-byte* percibido.                                       | Primer token < 3 s (red local)       |
| RNF-02 | Ejecución de código con timeout      | La ejecución en Piston dispone de un timeout HTTP de 30 segundos en el backend (`requests.post(..., timeout=30)`). Piston envía `SIGKILL` si el proceso del alumno excede su límite interno.                                                   | Timeout: 30 s (backend), configurable en Piston |
| RNF-03 | Paginación de datos                  | Las sesiones se paginan a 15 por página y los mensajes a 50 por página (máx. 100 configurable vía `?size=N`), evitando la transferencia masiva de datos y reduciendo la carga en PostgreSQL.                                                   | Sesiones: 15/pág, Mensajes: 50/pág   |
| RNF-04 | Ventana de moderación de output      | La moderación asíncrona de la respuesta del LLM se ejecuta cada `MOD_WORD_WINDOW` palabras (por defecto 20). Este parámetro es configurable vía variable de entorno para equilibrar latencia vs. cobertura.                                    | Ventana: 20 palabras (configurable)  |
| RNF-05 | Historial de contexto limitado       | El payload enviado al LLM incluye como máximo los últimos 10 mensajes de la sesión (`order_by('-created_at')[:10]`), limitando el consumo de tokens y mejorando el tiempo de respuesta.                                                       | Máx. 10 mensajes de contexto         |
| RNF-06 | Vigencia de tokens JWT               | El `access_token` tiene una validez de 1 día y el `refresh_token` de 7 días, equilibrando seguridad con comodidad de uso.                                                                                                                    | Access: 1 día, Refresh: 7 días       |

## Aislamiento y Contenedores Docker

| ID     | Nombre                               | Descripción                                                                                                                                                                                                                                  | Métrica / Restricción                |
|--------|--------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|--------------------------------------|
| RNF-07 | Ejecución aislada con Piston         | Todo código del alumno se ejecuta dentro del contenedor Docker de Piston (`ghcr.io/engineer-man/piston`), completamente aislado del host y del backend Django. El contenedor se ejecuta en modo `privileged` para gestionar sus propios sub-contenedores de ejecución. | Aislamiento completo por contenedor  |
| RNF-08 | Almacenamiento efímero en Piston     | Los directorios `/piston/jobs` y `/tmp` del contenedor Piston se montan como `tmpfs` (sistema de archivos en RAM), garantizando que no persista ningún artefacto de ejecución del alumno tras cada petición.                                   | tmpfs en `/piston/jobs` y `/tmp`     |
| RNF-09 | Base de datos contenerizada          | PostgreSQL 15 (Alpine) se ejecuta como servicio Docker independiente con volumen persistente `postgres_data`. El puerto se mapea en `5433:5432` para evitar conflictos con instalaciones locales.                                              | PostgreSQL 15-alpine, puerto 5433    |
| RNF-10 | IA local con Ollama                  | Los modelos LLM (tutor y moderación) se ejecutan localmente a través de Ollama, sin depender de APIs externas de pago. Esto garantiza privacidad total de los datos del alumno y elimina costes por token.                                     | Sin dependencia de servicios en la nube |

## Seguridad y Moderación

| ID     | Nombre                               | Descripción                                                                                                                                                                                                                                  | Métrica / Restricción                |
|--------|--------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|--------------------------------------|
| RNF-11 | Autenticación obligatoria            | Todos los endpoints de la API (excepto registro, login y refresh) requieren un token JWT válido (`IsAuthenticated`). Las peticiones sin token o con token expirado reciben HTTP 401.                                                           | JWT Bearer en todas las peticiones   |
| RNF-12 | Renovación transparente de sesión    | El frontend implementa un interceptor Axios que detecta errores 401, intenta renovar el `access_token` automáticamente y reintenta la petición original. Las peticiones concurrentes se encolan durante el refresh (patrón *subscriber queue*). | Auto-refresh transparente            |
| RNF-13 | Moderación dual configurable         | El sistema soporta cuatro modos de moderación configurables desde el Admin: `both` (input + output), `input` (solo entrada), `output` (solo salida) y `none` (desactivada). Esto permite ajustar el nivel de seguridad según el contexto de uso. | 4 modos: both, input, output, none   |
| RNF-14 | Fail-closed en moderación de input   | Si la moderación de entrada falla por error técnico, el mensaje se bloquea por defecto (estrategia *fail-closed*), priorizando la seguridad sobre la disponibilidad.                                                                          | Bloqueo por defecto en error         |
| RNF-15 | Fail-open en moderación de output    | Si la moderación de salida falla por error técnico, el texto se permite (estrategia *fail-open*), priorizando la continuidad del stream educativo sobre el riesgo residual.                                                                    | Permiso por defecto en error         |
| RNF-16 | Prompt anti-jailbreak                | El `SYSTEM_PROMPT` del tutor incluye reglas explícitas de *prevención de engaños*: ignora instrucciones del alumno que intenten anular las directrices del sistema, solicitar código completo o cambiar el rol del asistente.                   | Regla explícita en el system prompt  |
| RNF-17 | CORS configurado                     | En desarrollo se permite `CORS_ALLOW_ALL_ORIGINS = True`. En producción se restringe a los orígenes autorizados mediante `CORS_ALLOWED_ORIGINS`.                                                                                              | Configurable por entorno             |
| RNF-18 | Propiedad de datos por usuario       | Todas las consultas de sesiones y mensajes filtran por `user=request.user`, impidiendo que un usuario acceda a datos de otro. El borrado en cascada (`on_delete=CASCADE`) garantiza la eliminación completa.                                   | Filtrado por usuario en cada query   |
| RNF-19 | Validación de entrada estandarizada  | Todos los datos de entrada se validan mediante serializers de DRF (`ChatInputSerializer`, `ExecuteInputSerializer`), limitando el tamaño del `prompt` a 2000 caracteres y garantizando la sanitización de tipos.                               | Máx. 2000 chars en prompt            |
