# Tabla 3-1: Requisitos Funcionales de SocratiCode

| ID    | Nombre                                | Descripción                                                                                                                                                                                                                              | Prioridad |
|-------|---------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-----------|
| RF-01 | Registro de usuarios                  | El sistema permitirá a un usuario crear una cuenta proporcionando `username`, `email`, `password` y `re_password`. La contraseña se valida mediante los validadores estándar de Django (`MinimumLengthValidator`, `CommonPasswordValidator`, etc.). Endpoint: `POST /api/auth/users/` (Djoser). | Alta      |
| RF-02 | Inicio de sesión con JWT              | El sistema autenticará al usuario mediante credenciales (`username`, `password`) y devolverá un par de tokens JWT (`access` y `refresh`). El `access_token` tiene una validez de 1 día y el `refresh_token` de 7 días. Endpoint: `POST /api/auth/jwt/create/` (SimpleJWT).                    | Alta      |
| RF-03 | Renovación automática de token        | El frontend interceptará respuestas HTTP 401 y ejecutará automáticamente un `refresh` del token. Si el refresh falla, se redirige al usuario a la pantalla de login. Endpoint: `POST /api/auth/jwt/refresh/`.                              | Alta      |
| RF-04 | Gestión de perfil de usuario          | El usuario podrá consultar y editar su `username`, `bio` y `theme` (light/dark). También podrá cambiar su contraseña proporcionando la actual y la nueva (con confirmación). Endpoints: `GET/PATCH /api/auth/users/me/`, `POST /api/auth/users/set_password/`. | Media     |
| RF-05 | Creación de sesiones de chat          | El sistema permitirá crear una nueva sesión de chat vacía asociada al usuario autenticado. El título por defecto será `"Nueva conversación"`. Endpoint: `POST /api/chat/sessions/create/`.                                                 | Alta      |
| RF-06 | Listado de sesiones paginado          | El sistema listará las sesiones del usuario ordenadas por `updated_at` descendente, con paginación de 15 sesiones por página. Endpoint: `GET /api/chat/sessions/?page=N`. | Alta      |
| RF-07 | Interacción con el tutor socrático    | El usuario enviará un mensaje de texto (máx. 2000 caracteres), opcionalmente acompañado de `code_context`, `last_output` y `language`. El sistema generará una respuesta pedagógica usando un LLM en modo streaming (SSE, token a token). Endpoint: `POST /api/chat/`. | Alta      |
| RF-08 | Streaming SSE de respuestas           | La respuesta del LLM se transmitirá en tiempo real mediante Server-Sent Events (SSE). Cada chunk contiene un JSON con el campo `token`. Al finalizar, se envía el evento `[DONE]` junto con `session_id` y `session_title`.                | Alta      |
| RF-09 | Moderación de input                   | Si está habilitada, el sistema evaluará el mensaje del alumno mediante un LLM de moderación antes de invocar al tutor. Si el veredicto es `NO`, se devuelve una respuesta predeterminada y se marca el mensaje como `moderated=True`. La moderación es síncrona y aplica *fail-closed*.      | Alta      |
| RF-10 | Moderación de output (asíncrona)      | Si está habilitada, el sistema evaluará la salida del LLM tutor en paralelo cada `MOD_WORD_WINDOW` palabras (por defecto 20). Si un fragmento acumulativo es rechazado, se corta la generación, se descarta el texto y se sustituye por el mensaje moderado. Aplica *fail-open*.            | Alta      |
| RF-11 | Auto-naming de sesión                 | Cuando el usuario envía el primer mensaje de una sesión cuyo título es `"Nueva conversación"`, el sistema asigna automáticamente como título los primeros 50 caracteres del texto del mensaje y actualiza `updated_at`.                     | Media     |
| RF-12 | Renombrado manual de sesión           | El usuario podrá renombrar una sesión proporcionando un nuevo título (máx. 200 caracteres). Endpoint: `PATCH /api/chat/sessions/<id>/rename/`.                                                                                             | Baja      |
| RF-13 | Eliminación de sesión                 | El sistema permitirá al usuario eliminar una sesión de chat y todos sus mensajes asociados (borrado en cascada). Endpoint: `DELETE /api/chat/sessions/<id>/delete/`.                                                                        | Media     |
| RF-14 | Historial de mensajes paginado        | El sistema devolverá los mensajes de una sesión ordenados por fecha descendente, con paginación de 50 mensajes por página (configurable hasta 100 vía `?size=N`). Endpoint: `GET /api/chat/sessions/<id>/messages/`.                        | Alta      |
| RF-15 | Compilación y ejecución de código     | El sistema ejecutará código fuente del alumno en un sandbox aislado (Piston) soportando `python3`, `c` y `java`. Devuelve `stdout`, `stderr`, `exit_code` y `language`. Endpoint: `POST /api/compiler/execute/`. Timeout: 30 s.                            | Alta      |
| RF-16 | Detección de timeout en ejecución     | Si la ejecución del código recibe la señal `SIGKILL` (tiempo excedido), el sistema añade un mensaje de error descriptivo indicando posible bucle infinito.                                                                                  | Media     |
| RF-17 | Configuración global (Singleton)      | El administrador podrá configurar desde el panel de Django Admin los parámetros globales: modelo LLM principal (`llm_model`), modelo de moderación (`moderation_model`) y modo de moderación (`both`, `input`, `output`, `none`). Solo puede existir una fila en la tabla `SystemConfig`. | Alta      |
| RF-18 | Contexto de código en el chat         | El sistema inyectará automáticamente el código del editor y la última salida de ejecución como mensajes de contexto del sistema en el historial enviado al LLM, permitiendo preguntas contextualizadas.                                    | Alta      |
| RF-19 | Protección de rutas (Navigation Guard)| El frontend protegerá las rutas con `requiresAuth` (redirige a `/login` si no autenticado) y `requiresGuest` (redirige a `/` si ya autenticado).                                                                                         | Alta      |
| RF-20 | Respuestas de error estandarizadas    | Todas las respuestas de error de la API siguen un formato unificado `{ "error": "...", "details": {...} | null }` mediante un `custom_exception_handler` en DRF.                                                                          | Media     |
