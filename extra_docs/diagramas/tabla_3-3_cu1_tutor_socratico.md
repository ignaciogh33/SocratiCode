# Tabla 3-3: Caso de Uso CU1 — Interacción con el Tutor Socrático

| Campo              | Descripción                                                                                                                                                                                                                                           |
|--------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Identificador**  | CU1                                                                                                                                                                                                                                                   |
| **Nombre**         | Interacción con el tutor socrático                                                                                                                                                                                                                    |
| **Actor principal**| Alumno (usuario autenticado)                                                                                                                                                                                                                          |
| **Actores secundarios** | Ollama (LLM tutor), Ollama (LLM moderación)                                                                                                                                                                                                    |
| **Descripción**    | El alumno envía una pregunta al tutor socrático, opcionalmente acompañada del código de su editor y la última salida de ejecución. El sistema modera la entrada, genera una respuesta pedagógica en streaming (SSE) y opcionalmente modera la salida en paralelo. |
| **Precondiciones** | 1. El alumno está autenticado (posee un `access_token` JWT válido). <br> 2. El servicio Ollama está activo y el modelo LLM configurado en `SystemConfig` está disponible. <br> 3. Existe una sesión de chat activa o se creará una nueva automáticamente. |
| **Postcondiciones**| 1. El mensaje del alumno queda persistido en la tabla `Message` con `role='user'`. <br> 2. La respuesta del tutor queda persistida con `role='assistant'`. <br> 3. Si fue el primer mensaje, la sesión se renombra automáticamente (auto-naming). <br> 4. El campo `updated_at` de la sesión se actualiza. |

## Flujo Principal (Happy Path)

| Paso | Actor   | Acción                                                                                                                                                                                                  |
|------|---------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 1    | Alumno  | Escribe un mensaje en el chat y pulsa enviar. El frontend envía `POST /api/chat/` con `{ session_id, prompt, code_context, last_output, language }`.                                                   |
| 2    | Sistema | Valida los datos de entrada mediante `ChatInputSerializer` (prompt ≤ 2000 caracteres).                                                                                                                 |
| 3    | Sistema | Obtiene o crea la sesión de chat (`_get_or_create_session`). Si `session_id` es `null`, crea una nueva sesión.                                                                                          |
| 4    | Sistema | Persiste el mensaje del alumno en la base de datos (`Message.objects.create`, `role='user'`).                                                                                                           |
| 5    | Sistema | Ejecuta `_auto_name_and_touch`: si es el primer mensaje de usuario y el título es `"Nueva conversación"`, asigna los primeros 50 caracteres como título. Actualiza `updated_at`.                        |
| 6    | Sistema | Lee la configuración global (`SystemConfig.get()`) para determinar: `llm_model`, `moderation_model` y `moderation_mode`.                                                                                |
| 7    | Sistema | **Moderación de input** (si `moderation_mode ∈ {both, input}`): invoca a `moderate_input()` que envía el texto del alumno al LLM de moderación. El veredicto es `OK` → continúa.                        |
| 8    | Sistema | Construye el array de mensajes para Ollama: `SYSTEM_PROMPT` + contexto de código (si existe) + últimos 10 mensajes del historial.                                                                       |
| 9    | Sistema | Inicia el streaming asíncrono: `ollama.AsyncClient().chat(model=llm_model, stream=True)`. Abre una respuesta `StreamingHttpResponse` con `content_type='text/event-stream'`.                             |
| 10   | Sistema | Por cada token recibido, emite `data: {"token": "..."}` al frontend vía SSE.                                                                                                                            |
| 11   | Sistema | **Moderación de output** (si `moderation_mode ∈ {both, output}`): cada `MOD_WORD_WINDOW` palabras (20 por defecto), lanza un `asyncio.Task` que evalúa el texto acumulado hasta el momento.              |
| 12   | Frontend| El `ReadableStream` del navegador procesa cada línea SSE. El callback `onToken(token)` concatena el token al mensaje del asistente en la interfaz de usuario, creando el efecto de escritura progresiva. |
| 13   | Sistema | Al finalizar el stream, emite `data: {"session_id": N, "session_title": "..."}` y `data: [DONE]`. Persiste la respuesta completa como `Message(role='assistant')`.                                      |
| 14   | Frontend| El callback `onDone(sessionId, sessionTitle)` actualiza el estado de la sesión activa y el sidebar.                                                                                                     |

## Flujos Alternativos

| ID    | Condición                                          | Acción del sistema                                                                                                                                                                                                                                     |
|-------|----------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| FA-01 | Moderación de input rechaza el mensaje (`NO`)      | Se marca el mensaje del usuario como `moderated=True`. Se envía un stream con la respuesta predeterminada: `"Lo siento, no puedo procesar ese mensaje. ¿Puedes reformular tu pregunta?"` seguido de `[DONE]`. No se invoca al LLM tutor.               |
| FA-02 | Moderación de input falla por error técnico         | Se aplica *fail-closed*: el mensaje se trata como rechazado (se bloquea). Se devuelve la respuesta moderada predeterminada.                                                                                                                             |
| FA-03 | Moderación de output rechaza un fragmento (`NO`)   | Se interrumpe la generación del LLM (`break` en el bucle de streaming). Se emite `data: {"moderated": true, "response": "..."}` seguido de `[DONE]`. Se persiste el mensaje con `moderated=True` y el contenido de la respuesta predeterminada.         |
| FA-04 | Moderación de output falla por error técnico        | Se aplica *fail-open*: se permite continuar el stream sin interrupciones (`return True`).                                                                                                                                                                |
| FA-05 | Error en el LLM durante el streaming                | Se emite `data: {"error": "Error en motor IA: <detalle>"}` al frontend. El callback `onError(error)` muestra el error al usuario.                                                                                                                      |
| FA-06 | Sesión no encontrada (`session_id` inválido)        | Se devuelve HTTP 404 con `{"error": "Sesión no encontrada.", "details": null}`.                                                                                                                                                                          |
| FA-07 | Token JWT expirado o ausente                        | El frontend intercepta HTTP 401, intenta renovar el token automáticamente (`POST /api/auth/jwt/refresh/`). Si falla, redirige al login.                                                                                                                  |
| FA-08 | Validación de entrada falla                         | Se devuelve HTTP 400 con los errores de validación en formato estandarizado `{"error": "...", "details": {...}}`.                                                                                                                                        |
